# 한컴 수식 렌더러 특이사항(quirks)과 실측 절차

수식 엔진(빌트인 `LatexToHwpConverter` + vendored hwip)의 자동 검증은
스크립트 문자열까지만 보장한다. "한컴이 실제로 어떻게 그리는가"는 실측만이
답이다 — 이 문서는 실측으로 확정된 렌더러 특이사항과, 남은 미확정 항목을
확인하는 절차를 기록한다. 새 특이사항을 발견하면 여기에 추가하고, 대응
코드는 `lib/equation/hwpx_equation_tool.py`(변환)와
`lib/equation/hwp_script_parser.py`(검증 어휘/동의어 표)에 반영한다.

## 확정된 특이사항 (실측 근거)

- **`tint`/`dint`/`qint` 본문 미렌더** (2026-06-15 실측): 수식 편집기에선
  ∭로 보이지만 한글 '본문' 렌더러는 글자 그대로 노출한다. →
  `normalize_hwp_script()`가 `int` 반복(∫∫∫)으로 치환.
- **`oiint`는 한컴 비키워드**: 글자 노출. → 진입 전 `\oint` 정규화.
- **키워드는 대소문자 구분**: 원소 In(인듐)·Ta(탄탈럼)은 키워드 IN·Tau 와
  충돌하지 않는다. 반대로 같은 글자라도 케이스가 다르면 다른 의미일 수
  있다(delta δ vs DELTA Δ).
- **비인용 다글자 런은 그리디 렉싱으로 쪼개진다**: `I_{pivot}` 처럼 키워드
  (`pi`)를 품은 런은 πvot 로 깨진다. → 텍스트 첨자는 반드시 인용
  (`I_{"pivot"}`). `hwp_script_parser.lint_hancom_lexing()`이 위험을 검출.
- **hwip 의 n제곱근 표기 `^{n} sqrt {x}` 는 선행 항에 결합**:
  `2\sqrt[3]{8}` → `2^{3} sqrt {8}` = 2³√8 로 렌더(구조 동치성 검사로 발견,
  2026-07-12). → `\sqrt[` 포함 식은 hwip 을 우회.
- **`root N of {X}` 는 한컴 문법이 아니다** (1차 렌더 시트 S1-10·S2-87 실측):
  `root` 가 sqrt 동의어처럼 다음 원자에만 걸려 √N + 이탤릭 'of' + X 로
  깨진다. → normalize 가 `{}^{N} sqrt {X}`(빈 밑 위첨자 + sqrt)로 정규화.
- **`sup` 은 위첨자 연산자** (S2-51·S2-68 실측): `\sup`(상한)을 알몸 `sup`
  으로 내면 수식 전체가 빈 렌더로 증발. `inf` 는 ∞ 키워드라 `\inf`(하한)의
  의미가 바뀐다. → preprocess 가 `\sup`/`\inf` 를 인용 리터럴("sup"/"inf")로.
- **`lnot` 은 비키워드** (S2-92 실측): `\neg p` → 'ln otp' 로 쪼개져 렌더.
  → normalize 가 `lnot` → ¬ 글리프로 치환(알몸 유니코드 글리프는 정상 렌더
  — °C·· 등 실측).
- **리터럴 중괄호 `\{ \}` 증발** (S2-114 실측): 알몸 '{' 는 그룹 여닫이로
  먹혀 집합 표기 {x|x>0} 의 중괄호가 사라진다. → 빌트인이 인용 리터럴
  ("{")로 내고, 해당 입력은 hwip 우회.
- **`=>` 는 ⇒ 가 아니다** (S1-3 실측): '=' '>' 두 글자로 렌더. 겹줄
  화살표는 키워드 RARROW/LARROW/LRARROW (전부 정상 실측).
- **OVERBRACE/UNDERBRACE 2인자형(`OVERBRACE {라벨} {본문}`) 빈 렌더**
  (S2-60 실측): 올바른 문법 미확정 — 2차 프로브(A/B/C형)로 판정 예정.
- **비정형 중첩(행렬 셀 안 다행 분수 `matrix{ {a # b} over {c} }`) 빈 렌더**
  (S2-58 실측): 쓰레기 입력 내성 케이스로, 실전 산출물에선 미관측.
- **슬래시 분수화와 구조 키워드의 상호작용**: `X / \left(...\right)` 가
  `{X} over {LEFT}` 로, `X / \int...` 가 적분 기호만 분모로 삼키는 파손
  (퍼저로 발견, 2026-07-12). → LEFT 통짜 factor 인식 + 구조 키워드 피연산자
  차단 + 1인자 키워드(sqrt/vec…) 인자 오분리 방지.

## 한컴은 렌더러가 둘이고 방언이 다르다 (2026-07-13 확정 — 최상위 원리)

| | 문서 렌더러(파일 열 때) | 수식 편집기(넣기) |
|---|---|---|
| 글자 런 해석 | **부분문자열 탐욕 매칭** — `pivot`→πvot, `initial`→∈ial (after 실측) | **토큰 단위** — `pivot` 은 그냥 글자 |
| `"..."` | 리터럴 문자열로 렌더 | 따옴표를 일반 글자 취급, 내용 재토큰화(스마트 곱슬따옴표化) |
| 재직렬화 | 없음 | 낱자 재결합(`p i v o t`→`pivot`), `{}` 빈 밑 제거, `\|`→vert, ¬→LNOT, ⇋→RLHARPOONS, `_`/`^` 앞 공백 삽입 |

따라서 **키워드를 품은 라벨(pivot·initial·in…)과 n제곱근 `{}^{n}` 트릭은
'생성 직후엔 정상, 사용자가 그 수식을 편집(넣기)하면 파손'** — 이는
손으로 타이핑해도 똑같이 발생하는 한컴 자체 비일관이다(완전 해결 불가,
경감만 가능). 왕복 완전 생존이 실측된 것: 키워드(max·min·pi·approx…),
키워드 없는 단어(out·vap·kJ·mol), 알몸 한글, `LEFT{ RIGHT}`, LNOT,
RLHARPOONS(하픈), 백틱 `=`.

## 수식 객체 '속성' — 생성 직후 비정상 렌더의 원인 (P1, 2026-07-13 해결)

넣기 전후 XML 비교로 확인: 편집기는 수식 객체에
`version="Equation Version 60" font="HYhwpEQ" lineMode="CHAR"
baseLine=64~89(내용별 계산) sz=계산값` 을 부여하는데 우리 생성기는
`baseLine="0"`, font/version/lineMode 누락이었다 — 이것이 '모든 수식이
넣기 후에야 정상으로 보이던' 원인.

**속성 프로브 판정(사용자, 2026-07-13)**: 정준 속성을 준 B/C/E/G 행 =
넣기 후와 동일한 정상. 현행 속성 A/D/F = 비정상. sz 복제(C)는 불필요
(B로 충분). 또한 E(정준 속성+결합 라벨 `_{pivot}`)가 **새로 열 때 pivot
으로 정상 렌더** — `version="Equation Version 60"` 이 문서를 새로 열 때의
스크립트 방언을 토큰 단위로 바꾼다(탐욕 πvot 파손이 사라짐).

**적용**: `make_equation()` 이 정준 속성 전체를 기본 부여(baseLine 은
내용 기반 근사 68/72/86 — `_estimate_base_line()`). 모든 파이프라인의
수식이 '처음부터 넣기 후 상태'로 생성된다.

잔여 관찰: 넣기를 누른 '직후 같은 세션'에서는 πvot 로 보인다(사용자
실측) — 편집 커밋 직후의 임시 렌더는 구식 렉서를 쓰는 것으로 보이며,
저장 후 다시 열면 v60 방언으로 정상 복귀할 것으로 추정(전이 근거: after
상태 = v60 속성 + 결합 라벨 = 속성 프로브 E행과 동일 입력이고 E행은 새로
열 때 pivot 정상 판정. 직접 재오픈 실측은 화면 권한 미승인으로 미수행).

## πvot 근본 해결 — 빈 그룹 글자 분리 (2026-07-13, 한글 실측 확정)

**증상**: 키워드를 품은 첨자 라벨(pivot⊃pi, initial⊃in, sup)이 문서 렌더러의
부분문자열 최장 일치 렉싱에 걸려 πvot·∈·증발로 깨진다. 재오픈 후에도 지속
되며 v60 속성으로도 못 막는다(앞선 "속성이 방언을 바꾼다"는 추론은 오판 —
사용자 재오픈 실측으로 반증).

**낱자+공백이 안 되는 이유**(실측): `I_{ p i v o t }` 는 생성 직후엔 pivot
정상이지만, 학생이 그 수식을 편집(넣기)하면 편집기가 `I _{pivot}` 으로
재결합해 재오픈 시 πvot 로 깨진다(사용자 after 파일 실측).

**해결 = 빈 그룹 {} 전 글자 분리**: `p{}i{}v{}o{}t`. 빈 그룹은 화면에 안
보이는 그룹 경계라 렉싱을 끊고(computer-use로 수식 편집기 직접 조작해
`I_{p{}ivot}` → pivot 정상 렌더 확인), 편집기 재결합도 막는다. 최소 삽입
(위험 키워드 경계에만 {})은 삽입이 새 인접쌍을 만드는 함정이 있어 폐기 —
`initial` 을 in·it 로 끊으면 'ni'(한컴 owns 기호 ∋)가 생겨 `T_{i∃tial}` 로
깨졌다(실측). 전 글자 분리는 이 2차 파손을 구조적으로 없앤다. 구현:
`hwpx_equation_tool._dequote_word` → `_split_keyword_run` = `"{}".join(word)`.

**한글 실측 검증(computer-use, 6종)**: pivot·sup·in·initial·vap·cm 라벨을
EQ-LATEX 로 생성 → 한글에서 넣기 없이 열자마자 전부 원문대로 정상 렌더 확인.

**rm(로만체)은 해결책이 아니다**(실측): `I_{rm pivot}`·`I_{{rm pivot}}` 모두
여전히 πvot. rm 은 글꼴 전환일 뿐 렉싱을 못 막는다.

  편집 왕복 후에도 파손 경로가 없다.

## 편집기 왕복(넣기) 불안정 — 따옴표 리터럴 (2026-07-12 최대 발견)

문서 '초기' 렌더러는 `"..."` 를 리터럴 문자열로 그리지만, **수식 편집기는
넣기 시 따옴표를 일반 글자로 취급하고 내용을 재토큰화한다**(사용자 실측:
`"sup"` → `“”`(sup 이 위첨자 연산자로 소모), `"inf"` → `“∞”`, `>=` → ≧).
즉 따옴표가 든 수식은 학생이 한 번이라도 편집하면 깨진다. 반면 사용자의
실제 보고서(2503codex_단진자.hwpx)의 편집기 제작 수식 19개는 전부
무따옴표(키워드 단어 + 낱자 + 백틱 `=` 유지) — 이것이 왕복 안전 정준형이다.

대응(2026-07-12): `dequote_script()` 가 normalize 최종 단계에서 모든 인용을
안정 표기로 푼다 — 한글·키워드 없는 단어(kg·out·percent)는 알몸, 함수명
키워드(max·lim)는 알몸 키워드, 키워드 품은 단어(pivot⊃pi, in, sup)는 낱자
분해(`p i v o t`), 실제 단어 사이 공백은 `~`. 집합 중괄호는 인용 대신
`LEFT{ RIGHT}` 쌍(균형일 때, preprocess 센티널 ⟬⟭ 경유). 구조 문자가 든
본문만 인용을 유지한다(희귀).

## (구) 2차 프로브 표 — 판정 완료, 위 결과로 대체

이번 수정의 렌더 확인 + 미확정 항목. 시트 S1-1~12 를 보고 아래를 기록:

| 프로브 | 내용 | 기대 | 실측 결과 |
|---|---|---|---|
| S1-1 | `2 {}^{3} sqrt {8}` 정규형 | 2·∛8 (3이 2의 지수로 붙으면 안 됨) | (대기) |
| S1-2 | `2^{3} sqrt {8}` 대조군 | 2³√8 (잘못 붙는 형태 확인용) | (대기) |
| S1-3 | `"{" x \| x > 0 "}"` | {x\|x>0} 중괄호 보임 | (대기) |
| S1-4 | `LEFT{ … RIGHT}` | 스트레치 중괄호 보임(향후 업그레이드 후보) | (대기) |
| S1-5 | `¬ p` | ¬p | (대기) |
| S1-6 | `not p` | not 키워드 거동 확인 | (대기) |
| S1-7 | `"sup"_{…}`·`"inf"_{…}` | 업라이트 sup/inf + 첨자 정상 | (대기) |
| S1-8 | `det`·`gcd`·`arg`·`deg` | 함수명 정상(증발/노출 없음) | (대기) |
| S1-9~11 | OVERBRACE A/B/C형 | 어느 형이 그려지는지 판정 | (대기) |
| S1-12 | `A ⇋ B` 알몸 하픈 | 글리프 렌더 여부(되면 ⇌ 보존 개선 가능) | (대기) |

## 렌더 시트 실측 절차

1. 시트 생성 (의심 프로브 + 골든셋 전체 + 실전 코퍼스 = 한 문서):
   ```bash
   .venv/bin/python3 scripts/eq_render_sheet.py        # tmp/eq-render-sheet.hwpx
   ```
2. macOS 한글에서 열어 위에서 아래로 훑는다 (5–10분). 확인 포인트:
   - 영단어 글자 노출(`exists`, `lVert`, `intercal` 류)
   - 빈 박스/빈 분수 인자
   - 기호 어긋남(≤ 로 그려진 ⇐, 단줄로 격하된 ⇔ 등)
   - 라벨(전각 원문)과 수식 구조 불일치(분모/분자 뒤바뀜, 첨자 위치)
3. Windows 한컴에서도 릴리스마다 1회 반복(파일 열림 자체도 확인).
4. 발견 항목은 번호(S1-3 등)로 기록 → 위 표에 실측 결과 기입 →
   동의어 표/변환 표를 고치고 아래 게이트를 다시 돌린다.
