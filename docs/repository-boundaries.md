# 저장소 공개 경계

Quilo의 웹·Render 서버 파이프라인은 공개 저장소를 기준으로 개발합니다. 내부 운영
정보, 사용자 데이터, 재배포 권리가 없는 자산은 별도 비공개 저장소와 외부 비밀
관리자에 둡니다.

| 저장소 | 공개 범위 | 역할 |
|---|---|---|
| `fakeminjun7321/Quilo` | 공개 | 웹 UI, 서버, 보고서·문서 파이프라인, 공개 문서와 테스트의 기준본 |
| `fakeminjun7321/quilo-internal` | 비공개 | 과거 비공개 이력, 운영 메모, 공개할 권리가 없는 자산의 위치 기록 |

비공개 저장소는 공개 저장소의 개발 기준본이나 자동 미러가 아닙니다. 소스 수정은
가능하면 공개 저장소의 이슈와 PR에서 진행하고, 실제 비밀값은 어느 저장소에도
커밋하지 않습니다.

## 공개 저장소에 포함하는 항목

- 웹·Render 서버 소스와 정적 UI
- AI prompt, schema, 검증·후처리 코드
- 데이터베이스 schema와 재현 가능한 migration
- 인증·권한·크레딧·rate-limit 구현
- HWPX/DOCX/PDF 생성기와 파일 파서
- 합성 데이터로 만든 테스트와 공개 샘플
- 로컬 실행, 보안, 기여, 거버넌스 문서

공개된 구현은 곧 안전한 운영 기본값이라는 뜻은 아닙니다. 실제 배포자는
`SECURITY.md`, 환경변수 문서, migration, 외부 서비스 약관을 검토하고 자체
위협 모델과 운영 점검을 수행해야 합니다.

## 공개 저장소에서 제외하는 항목

- `.env`와 API 키, OAuth secret, 세션·암호화 키, 서비스 계정 자격 증명
- 사용자 업로드, 생성 보고서, 이메일·학번·계정 정보와 운영 로그
- 학교·기관 양식, 권리자가 공개 재배포를 허락하지 않은 HWPX/PDF/DOCX
- 비영리·사용 분야 제한처럼 OSI 정의와 맞지 않는 제3자 코드 및 파생물
- iPad·데스크톱 앱 소스와 서명·공증 자료
- 개인 장비 동기화 스크립트, 로컬 클라우드 경로, 작업 핸드오프 메모
- 홍보 영상의 음성·렌더·제작 작업 파일
- 미공개 취약점 상세, 실제 공격 payload, 사고 대응 자료

현재 제외 경로와 이유는 다음과 같습니다.

| 경로 | 이유 | 공개본 동작 |
|---|---|---|
| `lib/equation/vendor/` | 비영리 전용 제3자 변환 엔진 | builtin 변환기로 자동 폴백 |
| `public/equation/src/` | 위 엔진과 규칙의 파생 브라우저 구현 | 안내 페이지로 대체 |
| `lib/pipelines/*/templates/*.hwpx` | 학교·기관별 양식 재배포 권리 미확인 | 지원되는 파이프라인은 빈 문서 또는 명시된 fallback 사용 |

## 공개 스냅샷 규칙

과거 비공개 Git 이력에는 삭제된 비밀값이나 권리 제한 파일이 남을 수 있으므로 그
이력을 공개 저장소로 push하지 않습니다. 공개 저장소의 기존 공개 이력 위에 아래
절차로 정제된 현재 스냅샷만 반영합니다.

1. 비공개 저장소의 정확한 commit에서 격리된 worktree를 만든다.
2. 권리 제한 파일과 사용자·운영 데이터를 제거한다.
3. `npm run release:oss-check`로 파일 경계와 필수 정책 문서를 검사한다.
4. Gitleaks로 현재 스냅샷과 공개 저장소 이력을 검사한다.
5. 의존성 라이선스와 `THIRD_PARTY_NOTICES.md`를 대조한다.
6. `npm run verify:quick`, `npm run verify:core`, `npm run verify:release`의
   실제 통과 범위를 각각 기록한다.
7. 공개 저장소의 PR에서 파일 목록을 다시 검토한 뒤 merge한다.

## 새 제3자 코드·자산을 추가할 때

- 업스트림 URL, 버전 또는 commit, 저작권자, SPDX 라이선스를 기록한다.
- 상업·연구·교육 목적 제한이 있는 라이선스는 “오픈소스”로 분류하지 않는다.
- AGPL-3.0-or-later와 결합·배포할 수 있는지 확인한다.
- 폰트·샘플·템플릿도 코드와 똑같이 재배포 권리를 확인한다.
- 불명확하면 먼저 이슈에서 검토하고 merge하지 않는다.

라이선스 경계가 해소되지 않은 코드를 “일단 공개”하지 않습니다. 필요하면 기능을
분리하거나 독립 구현을 요청하고, 그 상태를 README와 이슈에 정직하게 표시합니다.
